# Pad Calendar Room 迁移 - 测试清单

Calendar 模块从纯 API 调用迁移到 Room DB + 本地 rrule 展开 + SyncStatus 离线优先架构（与 Task/Meal 模块一致）。本清单聚焦于 Event 的双向增删查改（Pad ↔ App），以及 repeat 按日/周/月/年的三种编辑/删除模式。

**架构变更：**

| 特性 | 迁移前（旧） | 迁移后（Room） |
| --- | --- | --- |
| 数据层 | 纯 API，无本地缓存 | Room DB + SyncStatus + 乐观更新 |
| 重复展开 | 服务端展开 | 本地 RRuleParser (`dmfs/lib-recur`) |
| 同步机制 | 每次请求全量加载 | Beat 触发增量同步 |
| 离线支持 | 无 | 完整离线 CRUD + 待同步队列 |
| 外部日历 | 服务端轮询 | Pad 端直接轮询（ICS/Google/Outlook Token Relay） |

**跨端测试说明：** 标记 **[App]** 的测试项需在 Flutter App 端验证数据同步。标记 **[Pad]** 的测试项从 App 端操作后在 Pad 端验证。UI 行为类测试仅在 Pad 端执行。

> **约定：** 每项测试应同时验证 Room DB 状态和后端 API 返回。"刷新" 指离开页面再返回，或杀掉应用重新打开，以确认数据持久化。

- [ ] 首次登录 → 分类 → 外部日历 → 事件按顺序同步到 Room
- [ ] 登出 → Room 数据库销毁
- [ ] 重新登录 → 全新同步
- [ ] **[Pad]** 验证初始同步后日/周/月/日程各视图数据正确

### 12.2 乐观创建

- [ ] 在线创建事件 → 立即出现在 UI 中，SyncStatus → SYNCED
- [ ] 离线创建事件 → 立即出现在 UI 中，SyncStatus → PENDING_CREATE
- [ ] 恢复网络 → 事件在下次心跳时同步，ID 更新为服务端 ID
- [ ] **[App]** 离线创建恢复网络后，App 端确认数据同步

### 12.3 乐观更新

- [ ] 在线编辑「所有事件」→ UI 立即更新，服务端调用随后完成
- [ ] 离线编辑 → UI 更新，SyncStatus → PENDING_UPDATE
- [ ] 恢复网络 → 更新同步

### 12.4 乐观删除

- [ ] 在线删除 → 事件立即消失，PENDING_DELETE → 硬删除
- [ ] 离线删除 → 事件立即从 UI 消失
- [ ] 恢复网络 → 删除同步

### 12.5 Beat 系统同步

- [ ] 在 App 修改事件 → Pad 心跳返回 `event=true` → 触发 `syncEventsFromServer()`
- [ ] 在 App 修改分类 → Pad 心跳返回 `event_cate=true` → 触发 `syncCategoriesFromServer()`
- [ ] 验证心跳轮询间隔（15 秒 TTL）正确触发刷新
- [ ] **[App]** Pad 修改事件 → App 通过自身 Beat 机制同步

### 12.6 待同步队列

> **注意**：并非所有操作都支持离线本地更新。编辑/删除「仅此事件」和「此事件及以后」**不做本地更新**（结果不可预测），而是等待网络恢复后发送到服务端再重新同步。仅「所有事件」编辑/删除和非重复事件支持完整离线乐观更新。

- [ ] 离线创建事件 → 同步前离线删除 → 验证服务端无孤儿记录
- [ ] 离线创建事件 → 同步前离线编辑 → 验证仅最终状态同步
- [ ] 多次离线操作 → 恢复网络 → 验证全部按正确顺序同步
- [ ] 离线编辑「仅此事件」→ 验证操作被挂起（UI 不立即反映修改），恢复网络后服务端处理并通过 Beat 同步回 Pad
- [ ] 离线删除「此事件及以后」→ 同上，恢复网络后同步

## 13. 重复编辑 + 删除组合操作

### 13.1 混合操作

- [ ] 创建每日重复事件 → 编辑「仅此事件」第 3 天 → 再编辑「所有事件」原始事件 → 验证第 3 天的独立事件不受影响
- [ ] 创建每日重复事件 → 删除「仅此事件」第 5 天 → 编辑「所有事件」改标题 → 验证第 5 天仍被排除
- [ ] 创建每日重复事件 → 编辑「此事件及以后」从第 4 天 → 删除「所有事件」**新**系列 → 验证第 1-3 天旧系列保留
- [ ] 创建每日重复事件 → 删除「仅此事件」第 2 天 → 删除「此事件及以后」第 5 天 → 验证第 1、3、4 天保留
- [ ] **[App]** App 端确认混合操作后数据一致

### 13.4 Exdate 格式归一化（定时事件 heartbeat 后 exdate 不失效）

> 服务端对定时事件存储 datetime 格式 exdate（如 `2026-03-14T10:00:00+00:00`），Pad 本地 rrule 展开用 date-only（`2026-03-14`）。Pad 用 `.take(10)` 归一化。此测试确保 heartbeat 同步后归一化仍然有效。

- [ ] 创建**定时**每日重复事件（如 10:00-11:00）→ 在 Pad 删除「仅此事件」某天 → 验证该天立即消失
- [ ] 等待 Beat 同步（服务端写回 datetime 格式 exdate）→ 验证该天**仍然被正确隐藏**（不会重新出现）
- [ ] 通过 curl/API 查看服务端 `repeat_exclude` 字段，确认包含 datetime 格式（如 `2026-03-20T10:00:00+00:00`）
- [ ] 在 Pad 日/周/月/日程视图中均确认被删除的实例不显示

### 13.2 边界日期

- [ ] 12 月 31 日每日重复事件 → 验证 1 月 1 日正确显示（跨年）
- [ ] 1 月 31 日每月重复 → 2 月、3 月 31 日、4 月 30 日 — 验证正确处理
- [ ] 2 月 29 日每年重复 → 验证非闰年行为
- [ ] UNTIL 设为今天 → 验证今天显示但明天不显示

### 13.3 首个实例边界情况

- [ ] 在首个实例上编辑「仅此事件」→ 验证 exdate 已添加，为该日期创建独立事件
- [ ] 在首个实例上删除「仅此事件」→ 验证 exdate 已添加，系列从第二个实例继续
- [ ] 在首个实例上编辑「此事件及以后」→ 验证原系列被完全替换为新系列
- [ ] 在首个实例上删除「此事件及以后」→ 等同于「删除所有事件」

## 14. 外部日历事件（只读）

> 外部日历事件（Google/Outlook/ICS）通过 `syncedCalendarId` 标记为只读，不可在 Pad 端编辑/删除。

### 14.1 ICS 外部日历

- [ ] 订阅 ICS URL → CalendarSyncWorker 轮询并导入事件
- [ ] 验证事件标记为只读（`isReadOnly = syncedCalendarId != null`）
- [ ] 验证 SHA256 变更检测：内容未变时跳过同步
- [ ] ICS 内容更新后 → 下次轮询检测到变化并更新 Room
- [ ] 验证事件正确显示外部日历颜色
- [ ] 点击外部日历事件 → 验证无法编辑（只读提示）
- [ ] ICS 中的**单日全天事件** → Pad 同步后验证仅显示 1 天（不是 2 天，exclusive end date 转换）
- [ ] ICS 中的**多日全天事件**（如 3天）→ Pad 同步后验证天数正确（不多 1 天）
- [ ] 在第三方日历中**编辑**重复事件的某个实例（改标题）→ Pad 下次轮询后仅该实例显示修改内容，其余不变（RECURRENCE-ID 异常处理）
- [ ] 在第三方日历中**删除**重复事件的某个实例 → Pad 正确隐藏该实例（exdate 生效）

### 14.2 Google Calendar（Token Relay）

- [ ] 已授权的 Google Calendar → Token Relay 获取 access_token
- [ ] 验证 token 5 分钟本地缓存：连续请求不重复调用服务端
- [ ] Token 过期 → 通过 Token Relay 自动刷新
- [ ] 事件使用哈希 ID 导入 → 正确显示
- [ ] 401 重试：清除缓存 → 获取新 token → 重试
- [ ] Google 中的**单日全天事件** → Pad 同步后仅显示 1 天（exclusive end date 转换）
- [ ] Google 中的**多日全天事件** → Pad 同步后天数正确
- [ ] 在 Google Calendar 中**编辑**重复事件某个实例 → Pad 同步后该实例显示修改内容，其余不变（recurringEventId 异常处理）
- [ ] 在 Google Calendar 中**删除**重复事件某个实例 → Pad 对应日期不再显示（cancelled instance → exdate）
- [ ] 验证 Google EXDATE（`EXDATE;VALUE=DATE:20260115,20260120`）被正确解析并跳过对应日期

### 14.3 Outlook Calendar（Token Relay）

- [ ] 已授权的 Outlook Calendar → Token Relay 获取 access_token
- [ ] 事件正确导入
- [ ] Token 过期 → 自动刷新
- [ ] Outlook 中的**单日全天事件** → Pad 同步后仅显示 1 天（exclusive end date 转换）
- [ ] Outlook 中的**多日全天事件** → Pad 同步后天数正确
- [ ] 在 Outlook 中**编辑**重复事件某个实例 → Pad 同步后该实例显示修改内容，其余不变
- [ ] 在 Outlook 中**删除**重复事件某个实例 → Pad 对应日期不再显示（cancelled exception 处理）
- [ ] Outlook 每周重复事件（BYDAY）→ Pad 显示在正确的星期几（不因 UTC 时区偏移一天）
- [ ] Outlook `recurrence` 格式正确转为 RRULE（验证 relativeMonthly 如"每月第二个周二"）

### 14.4 外部日历 Beat 同步

- [ ] 服务端外部日历变更 → `synced_cal` Beat 标志 → Pad 触发 `syncSyncedCalendarsFromServer()`
- [ ] 服务端新增外部日历 → Pad 自动发现并开始轮询
- [ ] 服务端移除外部日历 → Pad 停止轮询并清理 Room 中的事件

### 14.5 Pad 上传外部日历事件到服务端

> Pad 作为主同步器，同步外部日历后需通过 `uploadEventsToServer()` 将处理后的数据上传给服务端，供 App 和其他 Pad 设备使用。

- [ ] Pad 同步外部日历后 → 通过 API 查看服务端该 synced calendar 的事件数据，确认与 Pad Room 一致
- [ ] **[App]** Pad 同步外部日历后，App 端刷新确认也能看到这些事件（标题/时间/日期正确）
- [ ] Pad 同步外部日历的重复事件（含 modified instance）→ 验证服务端也存储了异常实例（`repeat_event_id` 指向父事件）
- [ ] 第二台 Pad 设备通过 `syncEventsFromServer()` 拉取数据 → 验证外部日历事件正确显示（如有条件）

## 15. 设备不匹配与错误处理

### 15.1 DEVICE_MISMATCH (403)

- [ ] 模拟 DEVICE_MISMATCH → 验证 `CalendarDatabaseDestroyed` 信号触发
- [ ] 数据库销毁 → EventManager 重新初始化并全新同步
- [ ] 重新同步后验证所有视图中数据正确

### 15.2 网络错误

- [ ] 创建时 API 超时 → SyncStatus → PENDING_CREATE，下次同步重试
- [ ] 更新时 API 超时 → SyncStatus → PENDING_UPDATE，下次同步重试
- [ ] 删除时 API 超时 → 重新插入为 PENDING_DELETE，下次同步重试
- [ ] 删除时返回 404 → 视为已删除，从本地 Room 移除

## 测试执行说明

- **后端要求**：所有测试需要运行 Django 后端 + Go 心跳服务
- **跨端说明**：**[App]** = 在 Flutter App 端验证；**[Pad]** = 在 App 端操作后，在 Pad 端验证
- **Beat 传播**：Beat 标志有 15 秒 TTL — 跨端同步最多等待 15 秒
- **离线测试**：在 Pad 设备上切换飞行模式
- **数据清理**：设置 → 销毁数据库，或登出/登录以获取干净状态
- **日期敏感**：部分测试涉及「今天」逻辑 — 记录测试执行日期
- **重复验证**：导航到多个日期验证模式 — 不要仅检查一天
- **视图验证**：对于重复事件，至少在日视图 + 另一个视图中验证
- **Room DB 检查**：使用 Android Studio Database Inspector 验证 Room 状态
